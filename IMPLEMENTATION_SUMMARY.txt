=== DECISION ENGINE (lib/udcf-core.ts) ===

Decision Algorithm:
1. CONSENT CHECK: Is user consent enabled for this data category?
   - NO → BLOCK immediately
   - YES → Continue to policy check
2. POLICY CHECK: Is this purpose authorized for this data category?
   - NO → BLOCK
   - YES → Continue
3. FINAL DECISION: ALLOW
4. LOG: Record decision with reason

Policy Rules:
- Analytics: [usage, analytics]
- Personalization: [profile, usage]
- Marketing: [profile]
- AI Training: [analytics, usage]

Logging:
- Every decision is logged
- Timestamps are immutable
- User IDs are masked for privacy
- Reasons explain the decision

=== DATA FLOW ===

User Workflow:
1. Login → User Dashboard
2. Set Consent (Profile: ON, Usage: ON, Analytics: ON)
3. Save Consent
4. View Access History (initially empty)
5. Later: Revoke consent
6. View Access History (shows BLOCK decision)

Application Workflow:
1. Login → Application Panel
2. Fill request form (user ID, purpose, data category)
3. Submit request
4. Receive ALLOW or BLOCK with reason
5. See decision logged in recent history

Admin/Regulator Workflow:
1. Access System Status
2. View real-time metrics
3. Review activity log
4. Check enforcement rules
5. Review audit trail for compliance

=== STORAGE ===

State Structure:
{
  consents: {
    "username": { profile: boolean, usage: boolean, analytics: boolean }
  },
  logs: [
    { id, timestamp, userId, appId, appName, purpose, dataCategory, decision, reason }
  ]
}

Persistence: localStorage (in-memory default, persists across refreshes)
No external database required for prototype

=== ARCHITECTURE HIGHLIGHTS ===

✓ Consent is centralized (users set once, firewall checks every time)
✓ Decisions are deterministic (same request → same decision)
✓ Policy rules are explicit (not hidden in code)
✓ Logs are immutable (no edit API)
✓ Roles are separated (users ≠ apps ≠ system)
✓ Audit trail is complete (every attempt recorded)
✓ Enforcement is transparent (reasons provided)
